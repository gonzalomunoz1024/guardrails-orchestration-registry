The big picture

What OPA and Rego actually do

OPA, the Open Policy Agent, is a tiny decision engine. Feed it a JSON object called input plus a policy file, and it hands back a decision. Rego is the language those policies are written in. You describe the conditions under which something is true, and the engine figures out the rest.

How OPA evaluates, in the abstract

A guardrail, in Rego

Rego is declarative. You describe what's true; OPA does the work. Hover any underlined token for a plain-English explanation of what that part of the policy is doing.

package approved_images.v1_0

default allow := false

# Allow only when the requested image comes from one of our Artifactory hosts.
allow if {
    image := input.document.spec.image.name
    some host in {"artifactory.com/", "artifactory-uat.com/", "artifactory-dev.com/"}
    startswith(image, host)
}

deny[msg] if {
    not allow
    msg := "Image must come from Artifactory"
}

Step 1 · Write

Author the policy locally

Open the studio, name the guardrail, choose its resource kind and enforcement. Start typing rego in the center pane. The live evaluator runs your policy against your sample document on every keystroke; the schema auto-derives from your document so contract and sample stay in sync.

Inside the studio

The Studio organizes a guardrail's lifecycle across six tabs.

Overview

At-a-glance metadata, evaluation stats, and a visual that explains exactly how the guardrail assembles its input: Document, Configuration, and any External dependencies all flowing into the rego policy.

Rego

The full read-only rego source as it was published. One click copies it. The published package carries a version-suffix (e.g. approved_images.v1_0) so multiple versions can live in the same OPA bundle without colliding.

Schema

The JSON Schema the guardrail publishes for the document it expects. This is the contract that suite adopters pin against. Auto-derived from your sample and refineable in a visual tree editor: toggle any field required or optional, set its type (string, object, array, …), or mark it any for fields whose shape varies between callers.

Tests

Validation against real traffic via regression test. Replay the guardrail against a window of historical inputs and see which requests would have been allowed, denied, or flipped from the prior version's decision.

Versions

Every published version of this guardrail, newest first. Each is immutable; once 1.0 is published, edits land as 1.1, 1.2, and so on. Suites pin against specific versions so a bump never silently changes what they enforce.

Configuration

The static configuration block the policy reads at input.configuration: allowed regions, approved images, off-hours windows. Published alongside the rego and resolved from the registry at evaluation time.

The Document Schema editor

Every guardrail publishes a JSON Schema describing the document it expects. The schema starts auto-derived from your sample and refines in a visual tree — no JSON wrangling required. Each row exposes the two knobs that matter most.

any Abstract fields

Some fields hold different shapes per caller (a free-form attributes blob, a per-app metadata bag). Marking the field any drops the type constraint so every caller can ship its own shape and still run through the same guardrail.

Required, per-parent

The required toggle controls whether the field is mandatory at its parent. A nested field can be required while its parent stays optional, or vice versa. The orchestrator validates against this on every inbound document.

Reserved fields stay safe

Orchestrator-injected fields (apiVersion, kind, metadata.appId, …) render with a 🔒 badge. You can't rename or retype them, but you can still mark them required for your own observability.

Step 2 · Test

Verify before you ship

Before you submit, the studio replays your edit against real historical traffic. You see the new rule and exactly which decisions it flips. No what-ifs.

The rule

Previously the policy accepted images from any of our Artifactory hosts (prod, UAT, or dev). The edit narrows it to the prod host only — UAT and dev images no longer qualify.

package approved_images.v1_0

default allow := false

    image := input.document.spec.image.name
    startswith(image, "artifactory.com/")  # was: any artifactory host

Step 4 · Publish

Immutable, by design

Every published guardrail version is locked the moment the pull request merges. The rego, schema, and examples are frozen at that (id, version) pair forever. Edit any of those three and a new version is born; the old one stays addressable for anyone still pinning to it.

Step 5 · Evaluate

Production enforcement

Guardrails don't run in isolation. They run as part of suites, grouped per workflow, pinned to specific versions. External services hit one endpoint; the orchestrator does the rest.

How a suite is built from guardrails

A suite is a curated bundle that runs together for a workflow. You pin specific versions; the suite stays mutable, the guardrails inside don't.

Guardrails are immutable. Suites pin versions. Re-pinning is an explicit upgrade.

Schema conflicts, caught at build time

Every guardrail in a suite receives the same input. If two members of the same resource kind declare contradictory shapes for a field, one of them will deny what the other accepts. The Suite Builder runs that check the moment you pin members — no waiting for production logs.

! Schema conflict · VirtualMachine

2 members in this suite target VirtualMachine but declare contradictory types for the same field. The orchestrator sends one input to all of them — one will deny what the other allows.

spec.size
vm_size_limit·string vm_cost_estimator·integer
metadata.appId
vm_size_limit·string vm_cost_estimator·number

Align the affected fields on each guardrail's Document Schema, or split them into separate suites. Mixed resource kinds (one VirtualMachine + one MongoDB) don't trigger this — they never see the same input.

How external services invoke a suite

Services like Lightspeed and VMForge don't call individual guardrails. They call the orchestrator with a document; the orchestrator looks up the matching suite and runs every applicable check.

One inbound document → orchestrator resolves the suite → runs every applicable guardrail → returns one rolled-up verdict.
